Introduction

This document defines the performance testing strategy for the DOM structured data ingestion and transformation workflow. The objective is to validate that the platform can process large healthcare datasets efficiently, reliably, and within agreed performance expectations.

The primary focus of this strategy is the backend data processing capability, including API-triggered pipeline execution, Azure Storage input datasets, manifest-based ingestion, Databricks processing, validation logic, quarantine handling, and target data availability in Unity Catalog or downstream tables.

This strategy is not intended to measure general user interface responsiveness. UI interactions may be used only as a supporting mechanism when required. Formal performance validation should be executed through backend APIs and monitored through Databricks, Spark UI, Azure monitoring, application logs, and test execution tools such as JMeter.

Current scope: structured data, primarily CSV-based ingestion. Unstructured data performance testing may be addressed in a separate strategy or future extension once processing rules and acceptance criteria are confirmed.

1.1 SLA Validation

Validate that ingestion and transformation meet the expected business performance requirements. The current baseline target is 10 million rows and 500 GB total dataset size processed in less than 4 hours. The 2x scenario targets 20 million rows and 1 TB total dataset size within the agreed degradation threshold.

Pending confirmation: the current interpretation is that the 2x scenario should not exceed approximately 20% additional processing time compared to the baseline target.

1.2 Scalability Assessment

  • Identify maximum validated dataset size.
  • Measure performance degradation as volume increases.
  • Determine whether processing time scales predictably.
  • Identify the volume threshold where SLA compliance is no longer achieved.
  • Identify technical limits before job failure, cluster instability, or excessive cost.

1.3 Concurrency Assessment

  • Determine whether executions run in parallel, sequentially, or through a hybrid model.
  • Measure the maximum sustainable concurrent pipeline executions.
  • Assess the impact of concurrency on processing time, Databricks jobs, cluster resources, storage I/O, scheduler behavior, and target table writes.

1.4 Stability, Reliability, Cost Efficiency, Recovery

Validate that large datasets can be processed without critical failures, excessive Spark spill, storage throttling, timeout failures, corrupted target data, or incorrect status updates. The test effort must also measure DBU consumption, estimate cost per run, and validate cleanup and recovery after heavy or failed executions.

2. Scope

2.1 In Scope

  • Structured CSV ingestion.
  • Manifest-based data ingestion.
  • API-triggered pipeline execution.
  • Databricks job/pipeline execution.
  • Data validation and quality checks.
  • Quarantine handling for invalid records.
  • Load, stress, and concurrency testing.
  • Spark, Databricks, Azure, and application-level monitoring.
  • Cleanup and recovery validation.

2.2 Out of Scope

  • Full browser/UI performance testing.
  • Self Service Portal upload performance testing.
  • Selenium or Playwright-based browser performance automation.
  • Unstructured data performance testing.
  • Production execution without explicit approval.
  • Security, penetration, access control, or compliance testing.

3. Key Assumptions

  1. Test datasets will be available in Azure Storage before pipeline execution.
  2. Performance testing will trigger ingestion through backend APIs or pipeline endpoints.
  3. The Operational Portal or backend services expose endpoints to create/configure pipelines, pass manifest or folder locations, trigger Databricks execution, and retrieve execution status.
  4. The baseline requirement applies to a full dataset/folder, not necessarily a single file.
  5. Test datasets may include one large file, many small files, or mixed file distributions.
  6. Databricks is expected to support parallel execution, but the actual concurrency model must be confirmed.
  7. The performance environment must be stable and sufficiently representative to produce meaningful results.
  8. Test data cleanup is required after execution to control storage costs.
  9. DBU consumption and execution metrics must be captured wherever technically possible.

5. Test Approach

5.1 Dataset-Based Testing

Performance scenarios should be defined by total dataset or folder size rather than individual file size only. This aligns with the expected pipeline behavior, where the process receives a manifest or folder location and processes all included files.

Baseline Dataset

Folder totaling 500 GB and approximately 10 million rows.

2x Dataset

Folder totaling 1 TB and approximately 20 million rows.

Edge Datasets

One 50 GB CSV, 50,000 small CSV files, and mixed-size datasets.

5.2 API-Based Execution

Formal performance testing should use API-based execution whenever possible. JMeter should be used to trigger pipeline creation/configuration, pass manifest or dataset locations, trigger Databricks execution, simulate concurrent execution, and capture API-level response times and failures.

5.3 Backend Monitoring

API response time alone is not sufficient. Backend execution metrics must be captured from Databricks, Spark, Azure, and application logs, including job duration, stage duration, retries, CPU/memory utilization, shuffle, spill to disk, storage throughput, write duration, cluster startup time, DBU consumption, and estimated cost.

7.2 Scenario Details

Baseline Structured Data Test

Input: 500 GB structured CSV, approximately 10M rows.

Success: completes under 4 hours, loads target data, handles invalid records, and updates final status.

Scalability Test — 2x Volume

Input: 1 TB structured CSV, approximately 20M rows.

Success: completes within agreed threshold without instability or excessive retries.

Incremental Volume Test

Run 500 GB, 1 TB, 2 TB, 3 TB, and higher volumes if approved to identify SLA threshold and production load recommendations.

Large File Edge Case

Validate processing of a dataset containing at least one 50 GB CSV without memory overflow or unacceptable Spark spill.

Small File Fragmentation

Evaluate behavior with approximately 50,000 small CSV files and measure enumeration, driver, metadata, and I/O overhead.

Data Quality and Quarantine

Measure validation overhead using 100% clean data and datasets with 1%, 5%, and 10% invalid records.

Concurrency Test

Trigger 2, 4, and 8 simultaneous executions, if approved, and capture start time, queue time, execution time, completion time, and final status.

Stress, Recovery, and Cleanup

Increase load until SLA breach or failure, validate recovery behavior, and confirm all test artifacts are cleaned up.

8. Positive and Negative Test Coverage

8.1 Positive Cases

  • Process clean structured CSV data.
  • Process 500 GB dataset within SLA.
  • Process 1 TB dataset within agreed scalability threshold.
  • Process datasets with mixed file sizes.
  • Process high file-count datasets.
  • Execute multiple pipelines simultaneously when supported by architecture.

8.2 Negative and Edge Cases

  • Corrupted or malformed records.
  • Missing headers and incorrect delimiters.
  • Invalid date formats and duplicate records.
  • High null percentage.
  • One very large file and 50,000 small files.
  • Concurrent execution beyond supported limits.
  • Resource-constrained execution, if approved.
  • Pipeline interruption followed by recovery validation.

10. Tooling Strategy

JMeter

Recommended for formal API-based performance execution, concurrent triggering, repeatable test plans, response time capture, error rate measurement, and execution reports.

Postman

Recommended for initial endpoint validation, authentication checks, payload confirmation, and small-scale exploratory execution.

Databricks and Spark UI

Used to analyze execution behavior, bottlenecks, resource utilization, stage delays, memory pressure, shuffle/spill behavior, failures, and retries.

Azure Monitoring

Used to track storage read/write activity, network throughput, infrastructure health, resource throttling, and cost indicators where available.

13. Open Questions

Business and Acceptance Criteria

  1. Does the 500 GB / 10M rows / < 4 hours SLA apply to one pipeline execution only?
  2. Does the 2x scenario mean exactly 1 TB / 20M rows?
  3. Does the 20% threshold mean the 1 TB scenario must finish within approximately 4 hours and 48 minutes?
  4. Is concurrency part of the formal acceptance criteria?
  5. How many concurrent clients or projects should be expected in production?
  6. What is the maximum expected dataset size per customer delivery?
  7. Should structured and unstructured data follow the same SLA?

Technical and Architecture

  1. Are Databricks pipelines executed in parallel, queued, or through a hybrid model?
  2. What is the maximum number of concurrent Databricks jobs allowed?
  3. Are there cluster, workspace, or job-level concurrency limits?
  4. Are there API rate limits or throttling rules?
  5. What timeout values exist at API, backend, Databricks, and storage layers?
  6. What happens when a pipeline fails?
  7. Is retry behavior automatic or manual?
  8. Are cleanup, delete, or restore endpoints available?

Test Data

  1. Who will generate the 500 GB and 1 TB datasets?
  2. Will test data be synthetic, anonymized production-like data, or real de-identified data?
  3. What schema should the CSV test files use?
  4. What percentage of invalid records should be included in negative tests?
  5. Should datasets include healthcare-like dummy data or generic test data?

Observability

  1. Who has access to Databricks and Spark UI during test execution?
  2. Where are pipeline execution logs stored?
  3. How is final pipeline status updated in the Operational Portal?
  4. Can DBU consumption be captured per run?
  5. Can storage throughput and throttling be monitored during execution?

Executive Summary

This strategy validates the structured data ingestion and transformation capabilities of the DOM platform using dataset-based performance testing. The primary baseline target is to process 500 GB / 10 million rows in less than 4 hours, followed by a 1 TB / 20 million row scalability test and additional stress, edge case, concurrency, recovery, and cleanup validations.

The testing approach prioritizes backend API-triggered pipeline execution, Databricks/Spark observability, cost measurement, and production-readiness recommendations. The final outcome will define validated processing limits, scalability behavior, concurrency capacity, cost implications, and recommended operating parameters for production.
